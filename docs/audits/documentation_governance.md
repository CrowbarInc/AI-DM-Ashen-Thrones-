# Documentation Governance

Date established: 2026-06-23

## Purpose

This policy governs audit-cycle documentation without treating every Markdown file as audit material.
The repository root is not an audit workspace. Human-authored audit records belong under
`docs/audits/`; generated outputs remain at their writer-owned paths; path contracts remain fixed
until an explicit migration changes every owner and validator together.

## Ownership Model

### Human-authored documentation

Human-authored discoveries, closeouts, plans, and decisions are maintained as prose. Their content is
the source of truth unless the document explicitly identifies a generator.

Owner responsibilities:

- choose the correct approved folder before creating the file;
- use a cycle/topic prefix and type suffix;
- link evidence rather than copying generated output into prose;
- update `audit_manifest.md` for a new cycle-level authority;
- archive superseded material without deleting traceability.

### Generated documentation

Generated documentation is owned by the script, helper, report writer, or refresh command that emits
it. The generator is the source of truth. Generated files must not be manually reorganized merely to
improve appearance.

Generated ownership includes:

- reports under `artifacts/`;
- checked-in reports regenerated by tools or test helpers;
- generated tables or files with explicit refresh commands;
- report families emitted together by one generator.

Changing a generated file's location requires changing the writer, its consumers, validation, and
documentation in one planned migration.

### Audit artifacts

Audit artifacts are evidence, not general project documentation. They may be machine-readable JSON or
CSV, rendered Markdown, test output, replay history, or diagnostic snapshots. Artifact paths may be
part of reproducibility and provenance; retain them where the producing workflow writes them.

### Test-owned documentation

Test-owned documentation is read, validated, named, or treated as authority by tests, fixtures,
replay registries, or CI. Tests define the path contract even when the file is prose.

Examples include:

- `docs/testing/protected_replay_manifest.md`;
- `docs/audits/cycle_k_block_k4_drift_threshold_policy_2026-05-26.md`;
- `audits/cycle_av_governance_registry_inventory.md`;
- `tests/test_inventory_governance.json`.

### Contract-owned documentation

Contract-owned documentation has an exact path referenced by production-adjacent tooling, validation,
refresh scripts, report readers, or CI. Test-owned files are a contract-owned subset, but a contract
may also be established by a generator or command-line workflow.

The owning reference—not the filename—determines whether the path is fixed.

## Approved Locations

| Documentation type | Approved location | Rule |
|---|---|---|
| Discovery and reconnaissance | `docs/audits/discovery/` | Human-authored cycle discovery only |
| Closeouts and final reports | `docs/audits/closeouts/` | Human-authored closure, readiness, and completion records |
| Metrics and scorecards | `docs/audits/metrics/` | Human-authored metric definitions and interpreted scorecards |
| Ledgers | `docs/audits/ledgers/` | Audit-specific longitudinal decisions and ownership ledgers |
| Evidence | `docs/audits/evidence/` | Deliberately retained human-readable evidence with no active writer-owned path |
| Scaffolds and instruction blocks | `docs/audits/scaffolds/` | Cycle scaffolds, execution checklists, and Codex instruction blocks |
| Generated reports | Writer-declared path, normally `artifacts/` | Do not relocate independently of the generator |
| Test and governance contracts | Existing declared path, commonly `docs/testing/`, `docs/audits/`, or `tests/` | Path remains fixed while referenced |
| Historical audit records | `docs/audits/archived/` for new archival decisions | Existing historical trees may remain in place when migration value is low |
| Ambiguous material | Existing path, documented in the manifest or closeout | Do not move until ownership is established |

Normal architecture, gameplay, system, user, and test-running documentation remains under its normal
project location. A filename containing `audit`, `contract`, `inventory`, or `report` is not by itself
authority to move it.

## Path Contract Rules

### A file may be moved directly when

All of the following are true:

- it is human-authored;
- no script, tool, test, fixture, CI workflow, registry, or report reader references its exact path;
- it is not emitted by a generator;
- it is not a canonical manifest or active governance authority;
- active documentation references can be updated safely;
- the move has a clear information-architecture benefit.

Use `git mv`, search for the old full path and filename, update active references, and validate before
considering the move complete.

### A file requires migration planning when

Any of the following are true:

- a generator writes it;
- tests or CI read or validate it;
- a tool uses it as input or output;
- multiple generated files form one report family;
- historical replay, recurrence, or baseline provenance names the path;
- the file mixes hand-authored policy with generated sections.

A planned migration must update the owner, every consumer, refresh instructions, validation, and
manifest entries atomically. Moving only the checked-in output is prohibited.

### A file must remain fixed when

- an active path contract exists and no migration is authorized;
- it is a writer-owned artifact;
- it is a protected replay, recurrence, baseline, or governance authority;
- changing the path would require modifying tests or production behavior outside the documentation
  task's scope;
- its only benefit would be cosmetic folder symmetry.

## Future Cycle Rules

BW/BX/BY/BZ/CA/CB/CC-style work must use the following pattern:

| Cycle output | Location | Example |
|---|---|---|
| Initial discovery | `docs/audits/discovery/` | `CC_feature_readiness_closeout_discovery.md` |
| Final closeout | `docs/audits/closeouts/` | `BZ_protected_replay_trend_window_2_closeout.md` |
| Block scaffold | `docs/audits/scaffolds/` | `BZ1_protected_replay_trend_window_2_scaffold.md` |
| Metric definition or interpreted result | `docs/audits/metrics/` | `BZ2_recurrence_movement_classification.md` |
| Audit ledger | `docs/audits/ledgers/` | `<CYCLE>_<topic>_ledger.md` |
| Generated evidence | Generator-owned path, normally `artifacts/<cycle-or-topic>/` | `artifacts/by4/semantic_mutation_attribution_closeout.md` |

Additional rules:

1. Create the destination folder path before beginning the cycle; do not draft cycle reports in the
   repository root.
2. A generated report must declare its writer or refresh command in code or in the report header.
3. A human closeout should link generated evidence and should not duplicate the evidence payload.
4. Add the cycle's principal discovery, closeout, and contract authorities to `audit_manifest.md`.
5. If a test or tool must reference a documentation path, document that path contract in the
   manifest or the owning file.
6. Do not create a second closeout when an existing closeout can be updated safely.

## Anti-Patterns

- Audit reports, scaffolds, or full-suite logs in the repository root.
- Generated outputs mixed into human-authored folders without identifying the writer.
- Manual edits to generated reports that will be overwritten on refresh.
- Moving a report while leaving its generator or tests pointed at the old path.
- Duplicate discovery or closeout documents for the same cycle and decision.
- Copying JSON/CSV evidence into Markdown instead of linking the authoritative artifact.
- Evidence files with no producing command, cycle/topic owner, or manifest entry.
- Treating ordinary architecture or gameplay documentation as audit material because its filename
  contains `contract`, `inventory`, or `audit`.
- Reopening historical organization work solely to achieve perfect folder symmetry.

## Governance Decision

Documentation cleanup is maintenance, not an evergreen feature stream. New violations should be fixed
when introduced. Existing generator-owned, contract-owned, and historical paths should remain stable
unless another task already needs to change their owner or consumer.
