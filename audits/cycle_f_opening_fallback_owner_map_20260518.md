# Cycle F.D Opening Fallback Owner Map

Date: 2026-05-18

Scope: recon-only ownership map for opening fallback. No runtime code, tests, comments, assertions, test locations, or `game/final_emission_gate.py` refactors were changed.

## Strongest Finding

Opening fallback is not a single owner. It is already split into four practical owners:

- `game/opening_deterministic_fallback.py`: semantic/composition owner for curated-facts-to-text fallback.
- `game/upstream_response_repairs.py`: upstream prepared payload owner.
- `game/final_emission_gate.py`: gate orchestration, selection, fail-closed, compatibility-local guard, and final output wiring owner.
- `game/final_emission_meta.py`: metadata and owner-bucket read mapping owner.

The drag is mostly not runtime ambiguity. The drag is repeated projection/triage assertions in replay, classifier, dashboard, and API tests that intentionally lock the same opening fallback facts from different consumer views. Future movement should stay recon-only until a human chooses whether those projection locks should remain duplicated, be comments-only clarified, or be relocated behind helper assertions.
